
## 事务的ACID

分布式事务的理论：CAP定理，BASE理论
C：一致性
A：可用
P：分区容错

## BASE是Basically Available（基本可用）、Soft state（软状态）和 Eventually consistent（最终一致性）三个短语的缩写

## 2PC：（Two-Phase Commit，两阶段提交）
	XA是其中一种实现
- prepare准备阶段: 写redo，undo日志，锁定资源，不提交事务

- 接受反馈ok/error: commit/rollback

优点：实现成本低，尽量保证数据的强一致，主流数据库都有自己的实现
缺点：单点问题，同步阻塞，数据不一致

## 3PC：（Three-Phase Commit，三阶段提交）
- canCommit: 校验，超时终止
- 正常进入preCommit阶段：执行事务，不提交，正常执行commit/失败或超时rollback
- 接受反馈：commit/rollback

3PC在2pc的基础上加了校验和超时判断
优点：
1. 超时机制解决资源占用一直阻塞问题
2. 加了一次校验，防止个别参与者不正常时，其它参与者执行事务锁定资源

缺点：
1. 用超时机制，导致同步阻塞，多了一次网络通信，性能变差
2. 没有解决数据不一致问题

## Seata
支持AT、TCC、XA、SAGA事务模式
- TC
- TM
- RM
TC 为单独部署的 Server 服务端，TM 和 RM 为嵌入到应用中的 Client 客户端

1.TM 请求 TC 开启一个全局事务。TC 会生成一个 XID 作为该全局事务的编号。XID，会在微服务的调用链路中传播，保证将多个微服务的子事务关联在一起。

2.RM 请求 TC 将本地事务注册为全局事务的分支事务，通过全局事务的 XID 进行关联。

3.TM 请求 TC 告诉 XID 对应的全局事务是进行提交还是回滚。

4.TC 驱动 RM 们将 XID 对应的自己的本地事务进行提交还是回滚。

AT模式前提：支持本地ACID事务，基于java的jdbc访问数据库
整体机制：
二阶段提交协议演变：
- 一阶段：undolog和业务数据在同一个事务中提交，然后释放锁和资源
- 二阶段：
	- 提交是异步的，非常快
	- 回滚根据一阶段的undolog进行业务数据的反向补偿

总的来说AT模式就是先提交事务，再通过undolog进行回滚反向--------+补偿

TCC：try-confirm-cancel，手动控制事务，可以支持不支持事务的数据库，手动控制回滚

- 空回滚问题，事务记录跟业务代码同时提交，回滚判断事务记录
- 幂等问题，执行过后修改事务记录表状态
- 悬挂问题，判断并记录悬挂状态，try去判断这个状态
```java
 public interface TccActionTwo {
    // 业务处理
    @TwoPhaseBusinessAction(name = "TccActionTwo ", commitMethod = "commit", rollbackMethod = "rollback")
    public boolean prepare(BusinessActionContext actionContext, int a);

   // 提交 
    public boolean commit(BusinessActionContext actionContext);

   // 回滚
    public boolean rollback(BusinessActionContext actionContext);
}
```

### Saga 长事务，老系统，封闭系统
在Saga模式中，所有的子业务都不在直接参与整体事务的处理（只负责本地事务的处理），而是全部交由了最终调用端来负责实现，而在进行总业务逻辑处理时，在某一个子业务出现问题时，则自动补偿全面已经成功的其他参与者，这样一阶段的正向服务调用和二阶段的服务补偿处理全部由总业务开发实现。

总结：事务的发起者来管理整个分支事务的执行和回滚，分支只管正常执行，然后提供本地事务，提供回滚接口
