# 一条 UPDATE 的完整 InnoDB 写入链路（终极串讲）

我们以执行：sql

```
UPDATE t SET name='xxx' WHERE id=1;
```

为例，从头到尾走一遍。

---

## 0. 前提知识（串讲必备）

- InnoDB 页：**16KB**
- 磁盘原子 IO：**4KB**
- 所以 1 页 = 4 次磁盘 IO
- 可能出现**页断裂（写一半宕机）**
- 解决方案：**WAL + Redo Log + DoubleWrite**

---

# 1. 先加载数据页到 Buffer Pool

1. 找到 id=1 所在的 **16KB 数据页**
2. 从 `.ibd` 文件读到内存 **Buffer Pool**
3. 如果已经在内存里，就直接用

这一步是**读**，还没写。

---

# 2. 在内存里直接修改数据（生成脏页）

1. 在内存里把 name 改成新值
2. 这个页在内存变脏 → **脏页（dirty page）**
3. **此时不会立刻写磁盘！**

这就是 MySQL 快的核心：**写内存，不写磁盘**。

---

# 3. 记录 Undo Log（用于回滚 & MVCC）

写在 redo 之前：

1. 先把**修改前的数据**记到 undo log
2. 用于事务回滚
3. 用于 MVCC 快照读

undo 也是日志，也会进 redo 保护。

---

# 4. 生成 Redo Log 写入 Redo Log Buffer（内存）

1. 把 “对某页做了什么修改” 记为一条 redo
2. 写入内存的 **redo log buffer**
3. 这一步**极快**，内存操作

redo 是**物理日志**：

“在 表空间 1 页 20 偏移 300 位置写入 字节 xx”

---

# 5. 事务提交：Redo 必须刷盘（WAL 核心）

`commit;` 时强制保证：

1. **redo log buffer → 操作系统缓存 → 磁盘物理落盘**
2. 必须是**顺序写**
3. 只有 redo 落盘成功，才算提交成功

这就是 **WAL（预写日志）**：

**先写日志，再改数据；日志先落盘，事务才算成。**

作用：

- 宕机不丢已提交数据
- 用顺序写替代随机写，性能提升几十倍

---

# 6. 后台异步刷脏页（随机写，不影响业务）

1. 后台线程（IO 线程、LRU 淘汰）
2. 把 Buffer Pool 里的脏页**刷到 .ibd 真实数据文件**
3. 这一步是**随机写**，慢，但不影响业务

问题来了：

**刷 16KB 页时可能写一半宕机 → 页断裂**

---

# 7. 刷脏页时必经：DoubleWrite 双写缓冲（防页损坏）

真正刷脏页到 ibd 前，必须走双写：

1. 把 16KB 脏页**先顺序写入双写缓冲区**（连续空间，快）
2. 确认双写落盘
3. 再**随机写入真实的 ibd 数据页位置**

### 为什么安全？

- 如果写真实页时宕机 → 页坏了
- 重启后，InnoDB 发现页损坏
- 直接从 **doublewrite 里拿完整 16KB 覆盖回去**
- 页恢复完整，再用 redo 继续恢复

双写作用：

**保证数据页物理完整，不让 redo 面对一个坏页没法恢复。**

---

# 8. 刷完脏页，推进 Checkpoint

1. 脏页真正落盘
2. 对应的 redo 就没用了
3. 推进 checkpoint，释放 redo 环形空间

redo 是循环写，不是无限增长。

---

# 9. 崩溃恢复时的流程（串起来才完整）

如果数据库突然宕机：

1. 启动先看页是否完整（双写兜底）
2. 从 checkpoint 位置开始重放 redo
3. 把内存里没刷盘的修改重新应用到数据页
4. 保证已提交事务 100% 不丢
5. 回滚未提交事务（undo）

---

# 终极一句话总结（面试满分）

**InnoDB 先在内存修改数据，通过 WAL 机制顺序写入 redo log 保证事务持久化；后台异步刷脏页时，通过 doublewrite 缓冲先写一份完整副本，防止 16KB 数据页因 4KB 原子 IO 导致页断裂，确保崩溃恢复时数据页完整可用，最终实现高性能 + 数据绝对安全。**
