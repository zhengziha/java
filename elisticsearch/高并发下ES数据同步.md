**高并发分布式系统里，MySQL 只做简单增删改，ES 只做复杂查询。数据从 DB 同步到 ES，主流就 3 种方案：双写、监听 Binlog、MQ 异步。**

下面用最实用、面试能直接说、项目能直接落地的方式讲清楚。

---

# 一、数据同步到 ES 的三种标准方案

## 1. 业务代码双写（最简单，小项目常用）

业务流程：

1. 写 MySQL
2. 直接调用 ES API 同步新增 / 更新 / 删除

优点：

- 最简单，不用中间件
- 实时性高

缺点：

- **高并发不推荐**
- 代码耦合，业务侵入强
- 事务不一致：MySQL 成功、ES 失败怎么办？
- 高并发下会拖慢接口

适用：

- 并发不高的后台系统
- 简单小项目

---

## 2. 基于 Binlog 同步（**企业级首选，高并发标配**）

流程：

1. 业务只写 MySQL
2. 用中间件监听 MySQL Binlog
3. 解析后自动写入 ES

常用工具：

- **Canal**（阿里，最常用）
- Debezium
- Maxwell

优点：

- 业务代码**零侵入**
- 异步、不影响主链路性能
- 高并发下非常稳
- 数据一致性好

缺点：

- 多部署一个组件
- 有毫秒级延迟

**这是你面试说 “高并发 + 异构查询” 的标准答案。**

---

## 3. MQ 异步同步（微服务常用）

流程：

1. 写完 MySQL，发送一条消息到 MQ（RabbitMQ/Kafka）
2. 消费者监听消息，异步更新 ES

优点：

- 解耦，不影响主接口性能
- 可重试、可堆积，适合高并发
- 微服务架构友好

缺点：

- 多一次 MQ 存储
- 增加架构复杂度

适用：

- 已经在用 MQ 的系统
- 需要高可扩展性的场景

# 二、ES 里数据怎么存？（结构设计）

高并发场景下，ES 结构有一个铁律：

**尽量存宽表，不做关联，不做 Join。**

做法：

1. 把需要查询的字段**全部冗余到 ES 文档里**
2. 一张订单，直接把用户信息、商品信息、店铺信息都放进去
3. 不使用 ES 的 join/nested/has_parent 这类关联查询（性能差）

示例结构：

json

```
{
  "orderId": 123,
  "userId": 456,
  "userName": "张三",
  "phone": "138xxxx",
  "productId": 789,
  "productName": "手机",
  "price": 4999,
  "createTime": "2026-01-01T12:00:00"
}
```

好处：

- 查询极快
- 高并发扛得住
- 分布式友好，不用跨库 Join

# 三、高并发下的关键注意点

1. **禁止实时关联查询**
    
    MySQL 联表、ES 联表都不要。
    
2. **数据允许短暂不一致**
    
    最终一致性即可，高并发系统不追求强一致。
    
3. **全量同步 + 增量同步**
    
    - 初始化：全量导入 MySQL → ES
    - 运行时：Binlog/MQ 增量更新
    
4. **避免频繁更新**
    
    ES 写多了性能会掉，高并发场景尽量**读多写少**。

# 四、面试怎么说最加分？

你可以直接背这段：

> 项目中 MySQL 负责核心事务存储，ES 负责异构查询，解决复杂条件、排序、分页、模糊查询等高并发场景。
> 
> 数据同步采用 **Canal 监听 MySQL Binlog** 异步同步到 ES，业务代码零侵入，不影响核心接口性能。
> 
> ES 存储采用宽表结构，冗余关联数据，避免在 ES 中做联表查询，保证高并发下查询性能稳定。

# 五、给你的建议

你要包装高并发经验：

- **选 Canal + ES 这套方案**
- 压测接口从 MySQL 联表查询改成 ES 查询
- QPS 轻松上几万，面试官非常认可
