# 热点方法加缓存优化 · 线程安全与数据一致性详解

> 由一次真实优化提炼：某游戏服务器 `GameTeam.getCharaList()` 被全项目 219 处高频调用，原实现每次都 `new ArrayList` + 逐成员查会话表，开销大。本文把它做成线程安全缓存的全过程，抽象成可复用方法论。
>
> 核心矛盾：**缓存 = 用"失效"换"不重算"。失效做不对，要么脏数据，要么并发崩。**

---

## 〇、一句话总纲

> **任何缓存正确性都建立在两条铁律上，缺一不可：**
> 1. **失效铁律**：凡是改动"源数据"的入口，都必须失效缓存。
> 2. **互斥铁律**：改动源数据的"写操作"，必须与"缓存构建"用同一把锁互斥。
>
> 光失效不够（失效可能被并发构建覆盖）；光加锁不够（漏一处失效就脏）。**失效定义"何时重建"，互斥保证"重建与失效不乱序"。**

---

## 一、先判断：这个方法该不该加缓存

加缓存前先过这 4 个条件，**任意一条不满足就要重新设计**：

| 条件 | 说明 | 本例 |
|------|------|------|
| **调用极频繁** | 热点路径，每秒/每 tick 多次 | 219 处调用，战斗/消息/匹配都在调 |
| **单次开销可观** | 有分配、查表、IO、或重复计算 | 每次 `new ArrayList` + 逐人 `getGameSession` 查 Map |
| **结果可由源数据推导** | 存在明确的"源 → 派生"关系 | `memberExList` + 在线状态 → 活跃成员列表 |
| **变更入口有限且可枚举** | 能找全所有改源数据的点 | 加人/踢人/暂离/归队/合并 |

⚠️ **特别注意"读方法带写副作用"**：本例 `getCharaList()` 原本会顺手 `removeGameTemDuiwu` 清理离线成员（读方法里藏了写）。缓存化时这种副作用只会在"未命中重建"时触发，频率骤降——必须确认是否有别的机制（如定时清理）兜底，否则行为会悄悄变化。

## 二、加缓存前必问的三个问题

### Q1：结果是否依赖"实时外部状态"？
如依赖"当前是否在线""当前时间"等会随时变化的状态，缓存必然有**一致性窗口**。要评估：
- 该状态变化时，是否有钩子能触发失效？（本例：断线 → `offline()` → `leaveTeam` → `removeGameTemDuiwu` → 失效，有兜底）
- 若无钩子，能否接受短暂脏数据？脏了后果多严重？

### Q2：源数据有几个变更入口？
**把所有写入点 grep 出来**，逐个确认是否封装在统一方法里。绕过封装直接改字段的地方（如 `teamB.getMemberExList().removeIf(...)`），缓存无法自动失效，要单独评估（本例那是解散中的死对象，可放过）。

### Q3：返回值会被调用方修改吗？
200+ 调用方里只要有一处 `.add/.remove/.clear`，就会污染共享缓存。
- grep 链式写操作（`getXxx().add(...)`）
- 仍要防"赋值后改"的隐性写 → **直接返回不可变视图**（`Collections.unmodifiableList`）做 fail-fast，比静默 corrupt 安全。

## 三、缓存的四种形态与选型

| 形态 | 适用 | 缺点 |
|------|------|------|
| **Lazy 计算 + 失效**（本例） | 读多写少，源数据变更入口明确 | 失效点必须埋全 |
| Eager 写时维护 | 写少且写路径集中，可顺便更新派生 | 写路径变复杂，易漏同步 |
| TTL 时间窗 | 强一致做不到、能容忍陈旧 | 必然有陈旧窗口 |
| 版本号/CAS 快照 | 无锁读要求极高 | 实现复杂 |

**选型原则**：优先 Lazy+失效。只有当失效点太多/太散埋不全时，才退而求其次用 TTL，并明确告知业务"这是最终一致"。

## 四、线程安全（核心实现）

### 4.1 经典双重检查锁（DCL）

```java
private volatile List<Chara> charaListCache;   // ① volatile 是必须的

public List<Chara> getCharaList() {
    List<Chara> cached = charaListCache;        // ② 命中：无锁 volatile 读
    if (cached != null) return cached;

    synchronized (this) {                        // ③ 未命中：加锁
        if (charaListCache != null) {            // ④ 双重检查（可能已被别线程构建）
            return charaListCache;
        }
        List<Chara> result = build();            // ⑤ 构建到【局部变量】
        charaListCache = unmodifiable(result);   // ⑥ 原子发布（volatile 写）
    }
    return charaListCache;
}
```

**每个步骤为什么必须：**
- **① volatile**：保证发布对其他线程立即可见，且禁止指令重排（避免"发布未初始化完的对象"）。
- **② 命中无锁**：热点快路径，必须不加锁，否则缓存白做。
- **③④ 双重检查**：防止多线程同时未命中时重复构建。
- **⑤ 局部变量构建**：**绝对不要边构建边往字段里 add**。否则并发读者（或递归调用）会读到半成品。
- **⑥ 原子发布**：构建完一次性 `volatile` 赋值。读者要么看到 `null`，要么看到**完整**列表，永远不会读到半成品。

### 4.2 "原子发布"为什么是关键

> 反例（半成品可见）：
> ```java
> // ❌ 错：直接往字段里 add
> for (...) charaListCache.add(...);   // 别的线程此刻读到一个只加了一半的 list
> ```

正确做法永远是：**先攒够，再一次性赋值发布。**

### 4.3 ⚠️ 关键教训：失效操作本身也要加锁

这是最容易踩的坑。**光写个 `volatile` 失效（`cache = null`）不够。**

**"发布覆盖失效"竞态推演**（设 `cache==null`，成员 X 当前状态=1）：

```
线程A (构建, 持 synchronized(this)):
  1. 快照源数据
  2. 读 X.status → 读到 1（B 还没写）
  3. result 加入 X
        ─── B 插进来 ───
线程B (改 status + 失效, 【未加锁】):
  4. X.status = 2
  5. cache = null            （失效）
线程A 继续:
  6. publish cache = [..., X, ...]   ❌ 覆盖了失效！X 已变更却留在缓存
```

**根因**：A 持锁，但 B 不持锁，所以 B 能挤进 A 的"读源数据"和"publish"之间。原子性被打破。

**正解**：把"改源数据 + 失效"整体放进 `synchronized(this)`，与构建用同一把锁：

```java
public void leaveTempTeam(int id) {
    synchronized (this) {                        // 与 getCharaList 同一把锁
        // 改 memberteam_status ...
        invalidateCharaCache();
    }
    // 发消息等副作用留在锁外，避免持锁 IO
}
```

> **记法**：构建和失效是"同一笔事务的两半"，必须进同一把锁，否则会互相穿插。

## 五、数据一致性（失效的完备性）

### 5.1 失效铁律
> 凡是改动源数据（本例：`memberExList` 或 `memberteam_status`）的方法，**必须**调失效。

### 5.2 失效点完备性排查法（实操）
1. `grep` 所有源字段的**写入**点（注意区分 `==` 读和 `=` 写）。
2. 逐个确认：是否在封装方法内？是否在锁内？是否调了失效？
3. 特别留意**派生状态字段**——比如 `memberteam_status` 这种"对象内部可变、影响过滤结果"的字段，它的写入点比"增删集合"更隐蔽。
4. 排查**绕过封装**的直接写入（`xxx.getInternalList().removeIf(...)`），这些缓存感知不到。

### 5.3 典型漏失效场景（本例真实踩到）
- **`addMemberEx` 写成 `charaList.remove(newChara)`**：新成员根本没进过缓存，这是 no-op，加人后缓存不更新。
- **`addGameTeamChara` 暂离归队分支**：`status=2→1` 但位置没变，原有逻辑只检测"重排"才清缓存，漏了"纯状态变化"。
- **外部路径预设状态**：`goBackTeam` 先把 `status=1`，再调 `addGameTeamChara`，后者检测 `statusChanged=false` → 漏失效。**教训：派生状态不要在外部预设，交给唯一封装入口处理。**

### 5.4 返回不可变视图
```java
charaListCache = Collections.unmodifiableList(result);
```
- 防止 200+ 调用方任何一处误改污染缓存。
- 若真有调用方需要改，会抛 `UnsupportedOperationException` **立即暴露**，比静默脏数据强。
- 不要为了"安全"返回拷贝——那就把缓存的性能收益还回去了。

## 六、性能提升点

| 手段 | 效果 |
|------|------|
| **命中走 volatile 读，无锁** | 热点快路径开销趋近于 0 |
| **避免每次 `new` 集合** | 降低 GC 压力（高频路径分配是隐形大头） |
| **预分配容量** `new ArrayList<>(expectedSize)` | 减少扩容拷贝 |
| **失效而非全量重算** | 只有源数据变了才重建 |
| **副作用降频** | 原本"每次调用都触发的清理/发消息"降为"失效重建时才触发" |

**权衡提示**：本例构建里仍有"自清理离线成员"的副作用，失效重建时若恰好有离线成员会多构建一次（O(N²)，N 极小可忽略）。一致性正确性优先于这点开销。

## 七、解决性能问题的通用思路（方法论）

> 不只适用于加缓存，是"拿到一个慢方法"后的标准动作。

### 第 1 步：测量，别猜
- 用 profiler / arthas / 计时打点定位**真正的热点**，别凭感觉优化。
- 区分：是**单次慢**，还是**调用次数多**？两者优化方向不同。
  - 单次慢 → 优化算法/数据结构/IO
  - 调用多 → 减少调用 / 加缓存 / 批处理

### 第 2 步：拆解"贵在哪"
| 开销类型 | 表现 | 对策梯度 |
|----------|------|----------|
| **分配/GC** | 每次 new、装箱、大对象 | 复用对象、预分配、缓存、对象池 |
| **查表** | Map/DB/远程查询 | 缓存结果、索引、批量 |
| **锁竞争** | synchronized 等待 | 缩小临界区、读写锁、无锁结构、分片 |
| **IO** | 网络/磁盘 | 批量、异步、缓存、零拷贝 |
| **算法** | 复杂度高 | 换数据结构/算法 |

### 第 3 步：按"性价比梯度"选手段
```
算法/数据结构优化  >  减少调用次数  >  加缓存  >  并发化  >  底层调优
（收益大、风险低）                                   （收益递减、风险递增）
```
- 先看有没有 O(n²) 能降到 O(n)，再谈缓存。
- 加缓存属于"中等收益、有一致性风险"，**必须配套失效方案**。

### 第 4 步：加缓存的标准 checklist
- [ ] 确认可缓存（见第一节 4 条件）
- [ ] 选 lazy / eager / TTL
- [ ] 字段 volatile
- [ ] 构建：局部变量 + 原子发布
- [ ] 命中路径无锁
- [ ] **grep 出所有源数据写入点，逐一加失效**
- [ ] **写操作与构建同锁**
- [ ] 返回不可变视图
- [ ] 原方法的读副作用（自清理等）有无兜底
- [ ] 回归一致性：构造"写并发读"场景验证

### 第 5 步：验证
- 功能回归：增删改查后读到的缓存是否正确。
- 并发回归：多线程同时读写，看有无脏读/重复构建/异常。
- 性能对比：压测前后 QPS / 平均耗时 / GC。

## 八、常见坑（Anti-patterns 速查）

| 坑 | 现象 | 正解 |
|----|------|------|
| check-then-act 不加锁 | 首次并发构建 → 重复元素 | synchronized + 双重检查 |
| 边构建边写字段 | 读到半成品 | 局部构建 + 原子发布 |
| 只失效不互斥 | 发布覆盖失效 → 脏数据 | 写操作进同一把锁 |
| 增量更新缓存漏过滤 | 直接 add 成员绕过状态过滤 | 统一失效，由构建重算 |
| 返回可变引用 | 调用方误改污染缓存 | 返回不可变视图 |
| 漏掉绕过封装的写入 | 某路径改了源数据，缓存不失效 | grep 全部写入点；尽量收口到封装方法 |
| 派生状态外部预设 | 封装入口检测不到变化 → 漏失效 | 派生状态只由唯一封装入口改 |
| 读方法带写副作用 | 缓存后副作用频率骤降 → 行为变化 | 确认有无定时清理兜底 |

## 十、可复用代码骨架

```java
// 通用"高频只读方法 + 线程安全缓存"骨架
private volatile List<Foo> fooCache;

public List<Foo> getFoos() {
    List<Foo> cached = fooCache;
    if (cached != null) return cached;              // 命中：无锁

    synchronized (this) {
        if (fooCache != null) return fooCache;      // 双重检查
        List<Foo> result = new ArrayList<>(expectSize);
        // ... 由源数据派生 result ...
        fooCache = Collections.unmodifiableList(result);  // 原子发布
    }
    return fooCache;
}

private void invalidateFooCache() { fooCache = null; }

// 任何改源数据的写方法：
public void mutate(...) {
    synchronized (this) {                            // 与构建同锁
        // ... 改源数据 ...
        invalidateFooCache();                        // 失效
    }
    // 副作用（发消息等）放锁外
}
```
