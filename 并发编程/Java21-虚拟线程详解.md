## 一、先说结论：虚拟线程解决什么问题？

虚拟线程（Virtual Thread）是 **Java 21 正式发布** 的轻量级线程能力，对应 **JEP 444**。

它解决的核心问题不是：
- 让单个任务跑得更快
- 让 CPU 计算性能翻倍

它真正解决的是：

> 当系统里存在大量 **阻塞型任务**（如数据库、HTTP、RPC、文件 IO）时，如何继续使用 **“一个请求一个线程”** 这种直观编程模型，同时避免传统线程数量太多带来的内存和调度开销。

简单理解：

| 线程类型 | 特点 |
|------|------|
| **平台线程（Platform Thread）** | 直接映射到 OS 线程，数量多了成本高 |
| **虚拟线程（Virtual Thread）** | JVM 管理的轻量线程，创建和切换成本低，可非常多 |

---

## 二、为什么传统线程会成为瓶颈？

传统 Java 并发模型通常是：

```text
一个请求 -> 一个线程 -> 线程执行期间如果发生阻塞 IO，就一直占着这个 OS 线程
```

这会带来三个问题：

### 1. 线程内存成本高

平台线程默认需要较大的栈空间，线程多了以后，光线程栈就会吃掉很多内存。

### 2. 上下文切换成本高

OS 线程切换需要内核参与，线程数一多，CPU 会把很多时间浪费在调度上。

### 3. 高并发下不得不引入复杂异步模型

为了解决“线程太贵”，大家常常转向：
- 回调
- CompletableFuture
- Reactor
- Netty 异步编程

这些模型性能很好，但业务代码会明显变复杂。

虚拟线程的目标就是：

> 用更轻的线程，把同步写法重新变成高并发场景下可接受的方案。

## 三、虚拟线程和平台线程的关系

### 1. 平台线程

```java
Thread thread = Thread.ofPlatform().start(() -> {
    // 平台线程：本质上绑定 OS 线程
    handleRequest();
});
```

特点：
- 一个 Java 线程基本对应一个 OS 线程
- 适合数量不多、长期运行的线程
- 成本相对高

### 2. 虚拟线程

```java
Thread thread = Thread.ofVirtual().start(() -> {
    // 虚拟线程：由 JVM 调度到少量载体线程上运行
    handleRequest();
});
```

特点：
- 数量可以非常多，通常是“按任务创建”
- 阻塞时可从载体线程上卸载
- 更适合大量短任务、IO 密集型任务

### 3. 一个关键概念：载体线程（Carrier Thread）

虚拟线程并不直接等于 OS 线程，它运行时会被 JVM 挂载到平台线程上，这个平台线程就叫 **载体线程**。

可以把它理解成：

```text
虚拟线程 = 业务任务
载体线程 = 真正执行任务的少量工作线程
```

运行关系大致如下：

```text
很多虚拟线程 -> JVM 调度 -> 少量载体线程 -> OS 线程
```

## 四、虚拟线程为什么能多？核心原理是什么？

### 1. 不是“少开线程”，而是“阻塞时让出载体线程”

传统平台线程一旦执行到阻塞点：

```java
socket.read();
```

线程会卡住，同时对应的 OS 线程也被占住。

而虚拟线程在很多 JDK 已适配的阻塞操作上，会发生：

```text
阻塞前：虚拟线程挂载在载体线程上执行
阻塞中：虚拟线程从载体线程卸载，载体线程去执行别的虚拟线程
阻塞完成：虚拟线程再被重新调度继续执行
```

这就让“阻塞”不再等于“占住一个昂贵的 OS 线程”。

### 2. mount / unmount

常见术语：
- **mount（挂载）**：虚拟线程正在某个载体线程上运行
- **unmount（卸载）**：虚拟线程因为阻塞而离开载体线程

这就是虚拟线程高并发能力的根源。

### 3. 适合什么类型的任务？

最适合：
- 数据库访问
- RPC 调用
- HTTP 调用
- 文件读写
- 大量等待型任务

不适合指望它解决：
- 纯 CPU 密集计算
- 大量长时间占用 CPU 的算法任务

因为 CPU 核数并没有变多，虚拟线程不会让 CPU 计算凭空提速。

## 五、虚拟线程 vs 线程池，到底是什么关系？

很多人第一次看到虚拟线程，会问：

> 既然虚拟线程这么轻，还需要线程池吗？

结论是：

### 1. 虚拟线程通常不强调“复用线程”

传统线程池的核心思想是：

> 平台线程太贵，所以要复用。

虚拟线程的核心思想是：

> 虚拟线程很便宜，所以更推荐“一任务一虚拟线程”。

所以 Java 21 最常见的写法是：

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> handleRequest());
}
```

这个执行器的语义不是“固定复用几个线程”，而是：

> 每个任务分配一个新的虚拟线程。

### 2. 线程池仍然有价值，但用途变了

线程池仍可能用于：
- 限流
- 背压
- 控制外部资源并发度
- CPU 密集型任务的并行控制

也就是说：

> 在虚拟线程时代，线程池更多是“并发治理工具”，而不是“节省线程创建成本的唯一手段”。

## 六、Java 21 中怎么创建虚拟线程？

### 1. 直接创建并启动

```java
Thread.startVirtualThread(() -> {
    // 中文注释：适合直接启动一个独立的虚拟线程任务
    System.out.println("当前线程是否为虚拟线程：" + Thread.currentThread().isVirtual());
});
```

### 2. 使用 Builder 风格创建

```java
Thread thread = Thread.ofVirtual()
        .name("vt-order-", 1)
        .start(() -> {
            // 中文注释：可以批量指定线程名前缀，便于排查问题
            System.out.println(Thread.currentThread());
        });

thread.join();
```

### 3. 使用 ExecutorService 批量提交

```java
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10; i++) {
        final int taskId = i;
        executor.submit(() -> {
            // 中文注释：每个任务都会运行在自己的虚拟线程中
            Thread.sleep(Duration.ofSeconds(1));
            System.out.println("任务完成：" + taskId);
            return null;
        });
    }
}
```

> `ExecutorService` 可被正常关闭，推荐作为批量任务提交的主流写法。

## 七、一个直观对比：平台线程池 vs 虚拟线程

### 1. 平台线程池写法

```java
ExecutorService executor = Executors.newFixedThreadPool(200);

for (int i = 0; i < 10_000; i++) {
    executor.submit(() -> {
        // 中文注释：如果这里发生阻塞，线程池中的平台线程会一直被占住
        Thread.sleep(1000);
        return null;
    });
}
```

问题：
- 线程数不能无限增大
- 阻塞任务多时，队列容易堆积
- 需要自己估算池大小

### 2. 虚拟线程写法

```java
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10_000; i++) {
        executor.submit(() -> {
            // 中文注释：阻塞时通常会卸载，不会长期霸占载体线程
            Thread.sleep(1000);
            return null;
        });
    }
}
```

优势：
- 写法更接近同步思维
- 不需要非常纠结线程池大小
- 更适合海量阻塞型任务

## 八、虚拟线程最适合哪些业务场景？

### 1. Web 请求处理

典型场景：
- 一个请求里查数据库
- 调多个下游服务
- 写缓存
- 落日志

这些步骤有大量等待时间，虚拟线程很适合。

### 2. 批量调用外部接口

比如：
- 批量查询订单状态
- 批量发通知
- 批量拉取第三方数据

每个任务都可能阻塞等待网络返回，这类场景很适合“一任务一虚拟线程”。

### 3. 传统同步代码改造成本高的系统

如果系统已经是大量同步写法：
- JDBC
- RestTemplate 风格调用
- 同步文件处理

那么引入虚拟线程，往往比整体迁移到响应式框架更平滑。

---

## 九、虚拟线程不适合什么场景？

### 1. CPU 密集型任务

例如：
- 大量加密计算
- 图像处理
- 大规模排序
- 复杂规则引擎计算

这类任务瓶颈在 CPU，不在阻塞等待。  
虚拟线程不能突破 CPU 核数限制。

### 2. 需要严格控制并发度的外部资源访问

例如数据库连接池只有 50 个连接：
- 你开 10 万个虚拟线程并不会变成 10 万个数据库连接
- 真正卡住的资源仍然是连接池

此时要控制的是：
- 数据库连接数
- 下游接口并发数
- MQ 消费速率

### 3. 依赖线程本地变量过多的老代码

虚拟线程支持 `ThreadLocal`，但如果代码中：
- 大量缓存大对象到 `ThreadLocal`
- 把线程当成长生命周期容器

那在虚拟线程场景下会很浪费内存，因为虚拟线程数量通常远多于平台线程。

## 十、一个必须掌握的坑：线程固定（Pinning）

虚拟线程并不是任何阻塞都能完美卸载。  
最关键的限制之一就是 **线程固定（Pinning）**。

### 1. 什么叫 Pinning？

如果虚拟线程在某些场景下发生阻塞，但又 **不能从载体线程卸载**，那么载体线程就会被一直占住。

这时虚拟线程的优势就被削弱了。

### 2. 常见 Pinning 场景

#### 场景一：在 `synchronized` 保护范围内发生阻塞

```java
synchronized (lock) {
    // 中文注释：如果在同步块里执行长时间阻塞操作，
    // 虚拟线程可能无法及时卸载，导致载体线程被固定住
    Thread.sleep(Duration.ofSeconds(5));
}
```

#### 场景二：执行 native 方法或 foreign function 时阻塞

例如：
- 某些 JNI 调用
- 某些底层库调用

### 3. 如何规避？

建议：
- 不要在 `synchronized` 代码块里做长时间阻塞操作
- 更倾向使用 `ReentrantLock`
- 把锁保护范围缩小
- 避免把数据库、HTTP、文件 IO 放在长同步块里

### 4. 一个重要认知

不是说 `synchronized` 不能和虚拟线程一起用，  
而是说：

> **“持锁 + 阻塞”** 这个组合，在虚拟线程场景下尤其要谨慎。

## 十一、虚拟线程会替代响应式编程吗？

不会简单“一刀切替代”。

两者本质上解决的是相似但不完全相同的问题。

### 1. 虚拟线程的优势

- 编程模型直观
- 同步代码可读性高
- 改造成本低
- 调试体验通常更好

### 2. 响应式编程的优势

- 对背压、流式处理支持更强
- 在某些超高吞吐链路中仍然很有优势
- 整个生态已经形成成熟模型

### 3. 实战建议

如果你的系统特点是：
- 业务逻辑以请求/响应为主
- 代码大量同步阻塞
- 团队更熟悉传统 Servlet / JDBC 风格

那么虚拟线程通常是非常好的选择。

如果你的系统特点是：
- 需要复杂流式编排
- 需要细粒度背压控制
- 已经深度使用 Reactor / Netty

则不一定要强行改成虚拟线程。

## 十二、和线程池、连接池、并发控制的关系

虚拟线程很容易引出一个误区：

> 线程数量变大了，是不是吞吐就一定变大？

答案是不一定，因为真正的瓶颈往往在别处。

### 1. 数据库连接池仍然是硬上限

如果连接池只有 50 个连接：
- 5000 个虚拟线程同时查库
- 最终仍只有 50 个任务能真正拿到连接

### 2. 下游服务的 QPS 限制不会消失

如果第三方接口最多支持每秒 200 请求：
- 你开再多虚拟线程也只是更快把对方打挂

### 3. 正确用法

虚拟线程负责：
- 降低线程成本
- 简化同步编程

你仍然需要：
- 限流
- 信号量
- 连接池
- 队列
- 熔断降级

一个常见控制方式：

```java
Semaphore semaphore = new Semaphore(100);

try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (Task task : tasks) {
        executor.submit(() -> {
            semaphore.acquire();
            try {
                // 中文注释：即使使用虚拟线程，也要限制外部资源的并发访问量
                callRemoteService(task);
            } finally {
                semaphore.release();
            }
            return null;
        });
    }
}
```

## 十三、实战示例：批量请求最典型

下面用一个“批量查用户信息”的例子说明虚拟线程的价值。

### 1. 同步写法 + 虚拟线程

```java
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    List<Future<UserProfile>> futures = new ArrayList<>();

    for (Long userId : userIds) {
        futures.add(executor.submit(() -> {
            // 中文注释：这里可以直接写成阻塞式调用，代码逻辑非常直观
            User user = userRepository.findById(userId);
            Account account = accountClient.queryAccount(userId);
            Coupon coupon = couponClient.queryCoupon(userId);
            return UserProfile.of(user, account, coupon);
        }));
    }

    List<UserProfile> result = new ArrayList<>();
    for (Future<UserProfile> future : futures) {
        result.add(future.get());
    }
}
```

优点：
- 写法接近串行代码
- 不需要层层回调
- 不需要显式编排复杂异步链

### 2. 这类场景为什么特别适合？

因为多数时间都花在：
- 等数据库
- 等网络
- 等下游响应

不是花在 CPU 计算。

## 十四、迁移建议：老项目怎么落地？

### 1. 优先选择 IO 密集模块试点

例如：
- 聚合查询接口
- 批量导出
- 报表生成
- 下游编排调用

### 2. 先替换执行器，再观测指标

很多场景不需要重写业务逻辑，只需要把原来的线程池替换为：

```java
Executors.newVirtualThreadPerTaskExecutor()
```

然后重点观察：
- 吞吐量
- 平均响应时间
- P99
- 载体线程占用情况
- 数据库连接池等待时间

### 3. 排查 Pinning

迁移时重点检查：
- 长时间 `synchronized`
- JNI / native 调用
- 老旧驱动或库是否适配

### 4. 不要一上来全量改造

正确姿势通常是：

```text
先局部试点 -> 压测验证 -> 观察资源瓶颈 -> 再逐步推广
```

## 十五、虚拟线程对框架生态意味着什么？

### 1. 对传统同步框架是利好

例如：
- Spring MVC
- JDBC
- MyBatis
- 同步 HTTP Client

这类同步风格框架在虚拟线程加持下，能以更低改造成本承载更高并发。

### 2. 对响应式框架不是“终结”

Reactor、Netty、WebFlux 仍有价值，尤其在：
- 极致吞吐
- 背压控制
- 流处理
- 非阻塞生态完整链路

### 3. 一个更现实的趋势

未来很多系统会变成：
- 业务层大量采用虚拟线程保持同步写法
- 底层网络框架继续保持高性能非阻塞实现

这两者并不矛盾。

虽然本专题聚焦虚拟线程，但 Java 21 里还有两个常被一起提到的能力：

| 能力 | 状态 | 说明 |
|------|------|------|
| **虚拟线程（Virtual Threads）** | 正式特性 | JEP 444 |
| **Scoped Values** | 预览特性 | JEP 446，替代部分 ThreadLocal 使用场景 |
| **Structured Concurrency** | 预览特性 | JEP 453，更适合把一组并发子任务当作一个整体管理 |

面试里常见追问：

> 虚拟线程已经正式了，但结构化并发在 Java 21 仍是预览特性。

这个结论要分清。

## 十七、面试高频问题

### 1. 什么是虚拟线程？

> 虚拟线程是 Java 21 正式提供的轻量级线程，由 JVM 调度到少量载体线程上运行。它的目标是让开发者继续使用“一请求一线程”的同步编程模型，同时降低大量阻塞任务带来的线程成本。

### 2. 虚拟线程和平台线程区别是什么？

> 平台线程基本一一映射到 OS 线程，成本高；虚拟线程由 JVM 管理，数量可以很多，阻塞时通常会卸载，不会长期占住载体线程，更适合 IO 密集型任务。

### 3. 虚拟线程能提升哪些场景的性能？

> 主要提升大量阻塞型任务的并发承载能力，比如数据库访问、HTTP/RPC 调用、文件 IO。它不擅长提升 CPU 密集型任务性能。

### 4. 为什么虚拟线程不等于“无限并发”？

> 因为真正的瓶颈往往不是线程，而是数据库连接池、网络带宽、下游服务 QPS、磁盘吞吐等外部资源限制。

### 5. 什么是 Pinning？

> Pinning 是指虚拟线程在阻塞时无法从载体线程卸载，导致载体线程被固定占用。常见于 `synchronized` 保护范围内的阻塞操作，以及某些 native/foreign 调用。

### 6. 虚拟线程会取代线程池吗？

> 不会完全取代。虚拟线程减少了“复用线程”的必要性，但线程池或并发控制手段仍需要用于限流、背压和资源治理。

### 7. 虚拟线程会取代 Reactor / Netty 吗？

> 不会绝对取代。虚拟线程更适合保持同步写法、降低复杂度；而 Reactor/Netty 在背压、流处理和某些极致吞吐场景下仍然有优势。

## 十八、选型建议

### 优先考虑虚拟线程的场景

- 同步阻塞代码很多
- IO 密集
- 接口聚合调用多
- 团队希望降低异步编程复杂度

### 谨慎评估的场景

- CPU 密集计算
- 强依赖 `ThreadLocal` 缓存大对象
- 大量 `synchronized + 阻塞` 代码
- 外部资源瓶颈远大于线程瓶颈

## 十九、一句话总结

> **虚拟线程不是让任务执行得更快，而是让“海量阻塞任务也能继续用同步写法”这件事重新变得现实。**

如果你记住一句最重要的话，就是：

> **虚拟线程擅长解决“等待很多”的并发，不擅长解决“计算很多”的并发。**
