# 分布式事务：2PC 与 3PC 详解

> 独立专题笔记，汇总入口见 [java学习笔记汇总](./java学习笔记汇总.md)

---

## 一、背景：为什么需要分布式事务？

单体应用里，一个 `@Transactional` 就能保证 ACID。  
微服务 / 分库分表后，一次业务可能跨多个数据库或服务，例如：

```
订单服务(DB-A) 扣库存 + 支付服务(DB-B) 扣款
```

无法再用本地事务保证「要么全成功，要么全失败」，于是引入 **分布式事务协调协议**，2PC / 3PC 就是其中的经典方案。

---

## 二、2PC（Two-Phase Commit，两阶段提交）

### 1. 角色

| 角色 | 英文 | 职责 |
|------|------|------|
| **协调者** | Coordinator (TM) | 统一指挥提交/回滚 |
| **参与者** | Participant (RM) | 各数据库/资源，执行实际事务 |

XA 规范就是 2PC 的标准实现（MySQL XA、Oracle 等均支持）。

### 2. 两个阶段

```
协调者(C)                    参与者1(P1)              参与者2(P2)
   |                            |                        |
   |-------- prepare ---------->|                        |
   |-------- prepare ----------------------------------->|
   |                            | 执行SQL，写日志，不提交  |
   |                            |                        | 执行SQL，写日志，不提交
   |<------- YES ---------------|                        |
   |<------- YES ----------------------------------------|
   |                            |                        |
   |  [全部 YES → commit]       |                        |
   |-------- commit ----------->|                        |
   |-------- commit ------------------------------------>|
   |                            | 提交，释放锁             | 提交，释放锁
```

**阶段一 Prepare（投票阶段）**
- 协调者向所有参与者发 `prepare`
- 参与者执行 SQL，写 undo/redo 日志，**锁定资源但不提交**
- 参与者回复 YES（能提交）或 NO（不能提交）

**阶段二 Commit / Rollback（执行阶段）**
- 全部 YES → 协调者发 `commit`，参与者提交并释放锁
- 任一 NO → 协调者发 `rollback`，参与者回滚并释放锁

### 3. 2PC 的问题（面试必答）

| 问题 | 说明 |
|------|------|
| **同步阻塞** | Prepare 后资源被锁，其他事务无法访问，直到 Commit/Rollback |
| **单点故障** | 协调者挂了，参与者一直等，资源长期锁定 |
| **数据不一致** | 协调者发完部分 commit 后宕机，部分提交、部分未提交 |
| **性能差** | 两次网络往返 + 长时间持锁，不适合高并发 |

**典型不一致场景**：

```
协调者决定 commit，给 P1 发了 commit 后宕机
P1 已提交，P2 还在等 → 数据不一致
```

## 三、3PC（Three-Phase Commit，三阶段提交）

3PC 在 2PC 的 Prepare 和 Commit 之间加了一个 **CanCommit** 阶段，并引入 **超时机制**，减轻 2PC 的阻塞和单点问题。

### 1. 三个阶段

```
协调者(C)                    参与者1(P1)              参与者2(P2)
   |                            |                        |
   |  [阶段一 CanCommit]        |                        |
   |-------- canCommit? ------->|                        |
   |-------- canCommit? --------------------------------->|
   |                            | 只检查状态，不执行SQL    | 只检查状态，不执行SQL
   |<------- YES ---------------|                        |
   |<------- YES ----------------------------------------|
   |                            |                        |
   |  [阶段二 PreCommit]        |                        |
   |-------- preCommit -------->|                        |
   |-------- preCommit --------------------------------->|
   |                            | 执行SQL，写日志，不提交  |
   |<------- ACK ----------------|                        |
   |<------- ACK -----------------------------------------|
   |                            |                        |
   |  [阶段三 DoCommit]         |                        |
   |-------- doCommit --------->|                        |
   |-------- doCommit ----------------------------------->|
   |                            | 提交                    | 提交
```

**阶段一 CanCommit（询问）**
- 协调者问：「你能参与这次事务吗？」
- 参与者只检查自身状态（连接、权限等），**不执行 SQL、不锁资源**
- 全部 YES 才进入下一阶段

**阶段二 PreCommit（预提交）**
- 与 2PC 的 Prepare 类似：执行 SQL、写日志，**不提交**
- 参与者回复 ACK

**阶段三 DoCommit（提交）**
- 协调者发 `doCommit`，参与者提交
- 若协调者超时未收到响应，参与者可 **根据超时规则自行决定**（相对 2PC 的改进）

### 2. 3PC 相对 2PC 的改进

| 改进点 | 说明 |
|--------|------|
| **降低阻塞范围** | CanCommit 阶段不锁资源，只有 PreCommit 后才锁 |
| **超时机制** | 协调者/参与者超时后可按规则继续，减少无限等待 |
| **参与者状态机** | 明确状态：初始 → CanCommit → PreCommit → Commit，便于故障恢复 |

**参与者超时策略（简化理解）**：
- 在 **PreCommit 之后** 长时间收不到协调者消息 → 多数实现会 **自动提交**（认为其他节点已成功）
- 在 **CanCommit 阶段** 超时 → 不会贸然提交

### 3. 3PC 仍存在的问题

| 问题 | 说明 |
|------|------|
| **实现复杂** | 三阶段 + 超时 + 状态机，工程落地少 |
| **网络分区下仍可能不一致** | 极端分区场景下，理论上的 edge case 仍存在 |
| **性能仍一般** | 三次网络往返，仍不适合互联网高并发 |
| **实际使用少** | 工业界很少直接用 3PC，更多用 Seata/TCC/消息最终一致 |

## 四、2PC vs 3PC 对比总表

| 对比项 | 2PC | 3PC |
|--------|-----|-----|
| **阶段数** | 2（Prepare → Commit） | 3（CanCommit → PreCommit → DoCommit） |
| **资源锁定时机** | Prepare 阶段就锁 | CanCommit 不锁，PreCommit 才锁 |
| **阻塞程度** | 高，Prepare 后长时间阻塞 | 相对较低 |
| **协调者故障** | 参与者可能永久阻塞 | 超时后可按规则决策 |
| **数据一致性** | 协调者宕机可能不一致 | 理论上更好，极端场景仍有问题 |
| **网络开销** | 2 次 RTT | 3 次 RTT |
| **实现复杂度** | 较低 | 较高 |
| **工业应用** | XA、部分数据库 | 几乎不用，偏理论 |
| **适用场景** | 强一致、低并发、跨库短事务 | 同左，但实际很少选 3PC |

## 五、和 MySQL 的关联（常考串联）

MySQL 内部 **redo log 与 binlog 的两阶段提交** 是单机上的 2PC 思想，不是跨服务的分布式 2PC：

```
1. redo log prepare
2. binlog 写入
3. redo log commit
```

目的是保证 redo 与 binlog 一致，崩溃恢复时不会丢数据或主从不一致。

## 六、工业界怎么选？

2PC/3PC 都是 **强一致、同步阻塞** 方案，互联网业务更常用：

| 方案 | 特点 | 代表 |
|------|------|------|
| **2PC/XA** | 强一致，性能差，阻塞 | Seata XA 模式 |
| **TCC** | 业务层 Try-Confirm-Cancel，最终一致 | Seata TCC |
| **AT 模式** | 无侵入，基于 undo log | Seata AT |
| **消息最终一致** | 高吞吐，允许短暂不一致 | RocketMQ 半事务消息 |
| **Saga** | 长事务 + 补偿 | Seata Saga |

**选型口诀**：
- 强一致 + 低并发 + 短事务 → **2PC/XA**
- 高并发 + 可接受最终一致 → **TCC / 消息 / AT**

---

## 七、面试简答模板（30 秒版）

> **2PC** 分 Prepare 和 Commit：Prepare 时参与者执行但不提交并锁资源，协调者收集投票后统一 Commit 或 Rollback。问题是同步阻塞、协调者单点、可能数据不一致。  
> **3PC** 增加 CanCommit 询问阶段和超时机制，Prepare 前不锁资源，参与者超时后可自动决策，降低阻塞，但实现复杂、三次网络开销，工业界很少用。  
> 实际项目更常用 Seata AT/TCC 或 MQ 最终一致，而不是裸 2PC/3PC。
